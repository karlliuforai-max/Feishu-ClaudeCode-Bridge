# 飞书回复上下文支持计划

> 状态：✅ 已实现（2026-06-19）。下文保留原始设计，并在变更点标注实际落地情况。
> 目标：让用户通过飞书「回复」功能引用上下文、图片或文件时，bridge 能自动读取被回复消息，并把其中的文本与附件一并交给 Agent。

---

## 〇、实现摘要（2026-06-19 落地）

- 新增 `_resolve_mentions()`：正文里的 `@_user_N` 不再整体删除，改为渲染成可读的 `@姓名`，保留「@的是本机器人 / 他人 / 别的机器人」这层信息；拿不到名字的占位符仍删掉，不漏出 `@_user_N`。`_parse_message()` 的 text / post 分支都改用它。
- 新增 `_replied_message_id()`：只读直接父消息 `parent_id`（不递归 root/thread）。
- 新增 `_fetch_message()`：用 bot 身份 `client.im.v1.message.get` 拉取被回复消息，包成可复用 `_parse_message()` 的轻量对象；失败一律 fail-soft 返回 `None`。
- 新增 `_build_agent_prompt()`：区分「用户本次消息」与「回复引用的上下文」，无引用时退回原 `_build_media_prompt` / 纯文本行为。
- `on_message()` / worker 接入：空消息闸门放行「仅回复引用」的情况；worker 内拉取被回复消息、下载其图片/文件、合并构造 prompt，处理完一并清理临时文件。
- 单测：`tests/test_bridge_core.py` 新增 9 个用例覆盖 @ 渲染、回复识别、拉取失败兜底、各类 prompt 组合。

---

## 一、背景

当前 bridge 已支持用户直接发送的文本、图片、文件和富文本消息：

- `text`：提取正文，并把 @ 渲染成可读的 `@姓名`（原为整体删除 @，已按需求改为保留「@了谁」）。
- `image` / `file`：下载到 `workspace/inbox` 后把本地路径写入提示词。
- `post`：提取富文本正文、内嵌图片和文件。

但当前处理范围仅限「当前触发 Agent 的这条消息」。如果用户在飞书里使用「回复」功能引用另一条消息，再补充一句「帮我整理这个」或「看一下这个文件」，bridge 目前不会读取被回复消息，也不会下载被回复消息里的图片或文件。

这会导致用户以为已经提供了上下文，但 Agent 实际只看到当前消息正文。

---

## 二、目标行为

### 2.1 用户体验

用户在飞书中可以这样使用：

1. 回复一条文本消息并 @ 机器人：
   - 用户当前消息：`@MacPig 帮我整理一下`
   - 被回复消息：一段原始想法或会议记录
   - Agent 应看到当前指令和被回复文本。

2. 回复一条图片消息并 @ 机器人：
   - 用户当前消息：`@MacPig 识别这张图里的内容`
   - 被回复消息：图片
   - Agent 应拿到图片本地路径。

3. 回复一条文件消息并 @ 机器人：
   - 用户当前消息：`@MacPig 读一下这个 PDF`
   - 被回复消息：PDF / Word / Excel 等文件
   - Agent 应拿到文件本地路径。

4. 回复一条富文本消息：
   - 被回复消息里可能同时有文字、图片、文件
   - Agent 应同时拿到文本上下文和附件路径。

### 2.2 提示词效果

给 Agent 的 prompt 应明确区分「用户本次消息」和「用户回复引用的内容」：

```text
用户本次消息：
帮我整理一下

用户通过飞书回复引用了以下消息：
<被回复消息文本>

引用消息中的图片/文件已下载到本机：
- /path/to/image.png
- /path/to/document.pdf
```

如果当前消息本身也带附件，则继续保留现有附件逻辑，并和回复引用附件合并展示。

---

## 三、非目标

本次不做以下内容：

- 不读取完整群聊历史。
- 不自动把整条回复链无限展开。
- 不改变现有 session 维度和上下文持久化策略。
- 不改变 Agent 的实际执行方式。
- 不支持语音转写、表情包识别等新媒体类型。
- 不把回复上下文长期写入独立数据库，只作为本轮 prompt 输入。

---

## 四、技术方案

### 4.1 消息事件中识别回复关系

在 `on_message(data)` 中解析当前消息时，额外检查消息对象里是否存在回复关系字段。

飞书消息事件/消息详情里通常会包含类似字段，具体字段名需要以 lark-oapi SDK 实际对象为准：

- `parent_id`
- `root_id`
- `thread_id`
- `message_id`

实施时不要硬猜单一字段，应做一个小函数集中解析：

```python
def _reply_ref_from_message(msg) -> ReplyRef | None:
    ...
```

建议优先级：

1. 如果有直接父消息 ID，读取直接父消息。
2. 如果只有 thread/root 信息，读取 root 或 thread 中相关消息。
3. 如果没有可用关系字段，则返回 `None`，保持现有行为。

### 4.2 拉取被回复消息

新增一个只读函数：

```python
def _fetch_message(message_id: str):
    ...
```

职责：

- 调用飞书 IM message get / batch get API。
- 返回可复用 `_parse_message()` 的轻量消息对象。
- 失败时 fail-soft：打印日志，并把可读错误作为上下文提示给用户或 Agent。

注意：

- 需要使用 tenant token，因为当前 bridge 主要使用 bot 身份处理群消息。
- 机器人必须有权限读取该消息所在会话。
- 如果被回复消息已删除、不可见或超出权限，不能影响当前消息处理。

### 4.3 复用现有解析与下载逻辑

被回复消息拿到后，不另写一套解析器，复用现有：

- `_parse_message(ref_msg)` 提取文本和附件。
- `_download_resource(ref_message_id, file_key, type, name)` 下载图片/文件。
- `_build_media_prompt(...)` 的逻辑可以拆分升级，支持「当前附件」和「引用附件」分别展示。

建议新增结构：

```python
@dataclass
class MessageContext:
    text: str
    attachments: list[dict]
    source_message_id: str
    source_label: str
```

当前消息和被回复消息都可以转换成 `MessageContext`，最后统一构造 prompt。

### 4.4 Prompt 构造

新增函数：

```python
def _build_agent_prompt(
    current_text: str,
    current_paths: list[str],
    replied_text: str = "",
    replied_paths: list[str] | None = None,
) -> str:
    ...
```

输出原则：

- 用户当前消息永远放在最前面。
- 被回复消息明确标注为「引用上下文」，避免 Agent 误以为这是用户本次新说的话。
- 附件路径按来源分组，减少混淆。
- 如果只有被回复附件、当前消息没有正文，也要生成可执行 prompt。

### 4.5 群聊触发规则

保持现有群聊规则：群聊中必须 @ 机器人才响应。

后续可考虑一个增强项，但不建议第一版就做：

- 当用户回复机器人上一条消息时，即使没有 @，也触发响应。

原因：这个行为更自然，但也更容易误触发，需要额外判断被回复消息发送者是否为 bot，并确认群聊噪音风险。

## 五、边界情况

### 5.1 被回复消息不存在或无权限

行为：

- 当前消息仍正常处理。
- prompt 中可追加一句：

```text
用户使用了飞书回复功能，但被回复消息读取失败，请根据当前消息回答；必要时请提示用户重新发送上下文。
```

### 5.2 被回复消息是空消息或不支持类型

当前 `_parse_message()` 不支持 `audio` / `sticker` 等类型。第一版继续沿用这个限制。

行为：

- 不阻断当前消息。
- 如有必要，提示「被回复消息类型暂不支持」。

### 5.3 附件过大

沿用 `MAX_ATTACHMENT_BYTES` 限制。

行为：

- 下载失败时不把路径传给 Agent。
- prompt 中说明哪个附件未下载成功。

### 5.4 多层回复

第一版只读取直接被回复消息，不递归展开。

原因：

- 可控、低风险。
- 避免 prompt 过长。
- 避免把历史无关内容带入。

后续可通过配置扩展：

```json
{
  "reply_context_depth": 1
}
```

### 5.5 回复的是文件消息但用户当前消息无正文

只要当前消息 @ 了机器人，且能读取到被回复文件，就应正常处理。

prompt 示例：

```text
用户通过飞书回复引用了一个文件，文件已下载到本机：
- /path/to/file.pdf

用户没有补充文字说明，请先解析/描述这个文件的内容，再等待进一步指示。
```

## 六、配置建议

第一版可以默认开启，不新增配置；因为这是用户明确使用「回复」功能提供上下文，符合直觉。

如果担心变更影响现有行为，也可以新增配置：

```json
{
  "reply_context": true,
  "reply_context_depth": 1,
  "reply_context_max_attachments": 5
}
```

推荐第一版最小配置：

- `reply_context`: 默认 `true`
- `reply_context_depth`: 固定为 `1`，暂不暴露也可以

## 七、实现步骤

1. 调研并打印真实飞书回复事件对象字段。
   - 在安全不泄露 token 的前提下，临时输出 `msg` 的可见字段名。
   - 确认 SDK 对象里的回复关系字段。

2. 新增回复关系解析函数。
   - 输入：当前 `msg`
   - 输出：被回复消息 ID 或 `None`

3. 新增消息拉取函数。
   - 使用 bot 身份调用飞书 IM API。
   - 返回可传给 `_parse_message()` 的消息对象。

4. 扩展 prompt 构造。
   - 保留当前消息附件逻辑。
   - 增加被回复消息文本和附件分组。

5. 接入 `on_message()`。
   - 在进入 worker 前解析回复关系。
   - 在 worker 内下载当前附件和被回复附件。
   - 统一构造 prompt 后调用 `ask_agent()`。

6. 补测试。
   - 单测覆盖文本回复、图片回复、文件回复、富文本回复、读取失败、附件超限。

7. 真实群聊冒烟。
   - 回复文本消息。
   - 回复图片消息。
   - 回复 PDF 文件消息。
   - 回复富文本 post。

## 九、风险与控制

| 风险 | 影响 | 控制 |
|------|------|------|
| 事件字段名与预期不同 | 无法定位被回复消息 | 先打印字段名并用真实事件确认 |
| bot 无权限读取被回复消息 | 上下文缺失 | fail-soft，不影响当前消息 |
| prompt 过长 | Agent 响应慢或超上下文 | 第一版只读直接父消息 |
| 群聊误触发 | 群内噪音 | 第一版保持必须 @ 机器人才响应 |
| 文件过大 | 下载失败或耗时 | 沿用 `MAX_ATTACHMENT_BYTES` |
| 引用上下文与当前问题混淆 | Agent 答非所问 | prompt 中明确标注来源 |

---

## 十、建议第一版验收标准

第一版完成后，应满足：

- 回复文本消息时，Agent 能看到被回复文本。
- 回复图片消息时，Agent 能拿到图片本地路径。
- 回复文件消息时，Agent 能拿到文件本地路径。
- 当前消息自己带附件时，现有能力不回退。
- 被回复消息读取失败时，不影响当前消息正常处理。
- 群聊仍然只有 @ 机器人时才响应。
- 所有新增逻辑有单元测试覆盖。

---

## 十一、后续扩展

可以在第一版稳定后再考虑：

- 回复链多层展开。
- 回复机器人消息时免 @ 触发。
- 自动读取同一 thread 中最近 N 条上下文。
- 对引用图片做 OCR 前处理。
- 对引用 PDF / Word 自动抽取摘要后再传给 Agent。
- 在回复中标注附件原始文件名、大小、消息发送者和发送时间。
